## 1. 서론 (Introduction)

### 1.1 목적 (Purpose)
본 문서는 **"개발자가 셀렉터를 짜지 않아도, 사용자가 화면에서 본 값 한 줄만 주면
그 사이트의 목록을 구조화 데이터(CSV)로 추출"** 하는 크롤링 엔진의 요구사항과,
**기존 크롤링 방법 대비 차별점**을 기술한다. 제안서를 겸한다.

v3.0의 핵심 진전: 단순 추출을 넘어 **① 안티봇/로그인 사이트까지 실제 브라우저 세션으로
수집, ② 사이트마다 다른 '수집 성격'(로드방식·대기·페이지)을 레시피에 학습, ③ 한 번
성공하면 입력 없이 반복·스케줄 재현** 하는 "운영 가능한 크롤러"로 성숙했다.

v4.0의 핵심 진전: 목록만이 아니라 **④ 1차 목록 CSV의 링크를 따라 상세페이지까지 2단계로
수집**(체인 크롤링). URL 정련·iframe 인라인·본문/CSS 지정·미디어 폴백·랜덤 대기로, 흩어진
상세 정보(공고 본문 등)를 목록에 붙여 완결된 데이터셋을 만든다.

v5.0의 핵심 진전: **⑤ 값싼 방법이 다 실패해도 스스로 배운다** — save_as 로 로컬에 받아둔
HTML을 오프라인 LLM이 통째로 분석해 사용자가 지정한 필드를 재발견한다(라이브 사이트엔 추가
접속 0 = 탐지 회피). **⑥ 성공을 '원한 필드를 실제로 가져왔는가'로 재정의** — 다층 가드(필드
커버리지·차단 페이지 감지·의미 형태·LLM 백스톱)가 '채웠지만 쓰레기'(인증창·빈 페이지)를
성공으로 오인해 좋은 레시피를 덮는 사고를 막는다. **⑦ 저장 방식 4종 + 회차**로 누적/중복제외/
교체/키갱신을 사용자가 처음부터 고르고, **⑧ 실패를 개발자용 진단 저널에 남겨** 값싼 휴리스틱
자체를 개선한다. 이로써 웹메일(로그인)·커뮤니티 게시판(안티봇)를 LLM 치유 없이 save_as 만으로 실증했다.

### 1.2 범위 (Scope)
**시스템명**: 자가 치유형 웹 데이터 추출 엔진

이 시스템은 다음을 수행한다:
- 사용자가 입력한 **예시 값**(보이는 대로)을 단서로 HTML에서 반복 레코드와 각 필드의
  위치를 **역설계**(programming-by-example)
- 클래스명 난독화·구조 변경에도 **구조적 불변성**으로 계속 동작, 깨지면 **LLM이 그
  부분만 재인식**하여 자가 치유
- **정적 → JS 렌더링(SPA) → 안티봇 차단/로그인 사이트**까지 사이트별 하드코딩 없이
  하나의 흐름으로 대응 (로드방식을 사이트마다 자동 판단·기억)
- 페이지네이션 자동 순회·중복 제거, 결과를 **CSV로 누적 저장**, 파서·로드방식·대기시간을
  **레시피(CSV)로 저장하여 입력 없이 재현**, 성공 이력을 **감사 로그(_runs.csv)** 로 기록
- 과거 성공 크롤링을 **일괄 재현(replay)** → 윈도우 스케줄러로 정기 자동 수집
- **(v4.0) 체인 크롤링**: 1차 수집한 목록 CSV의 '링크 열'을 따라 각 **상세페이지**를
  **단일 레코드**로 추가 수집(2단계 크롤). 목록엔 없는 상세 필드(본문·급여·마감 등)를 붙인다.

**현재 상태**: 운영형 MVP. 12개 이상 실사이트에서 동작 검증. 정적/SSR·JS-SPA는 물론
**Akamai(이커머스)·DataDome(항공·여행 검색) 안티봇, 클래스 균일 SPA(소셜/트렌드 사이트), 로그인 세션이
필요한 페이지**까지 실제 추출 성공. **(v4.0) 채용정보 사이트 목록 32건 → 상세 본문 체인 크롤링**
(iframe 본문·이미지 공고 폴백 포함)을 실제 검증. **(v5.0) 웹메일 받은편지함 99건·커뮤니티 게시판
50건을 save_as로 돌파**(auto는 인증창에 막힘) — 접근만 뚫리면 값싼 결정적 엔진이 LLM 치유
없이 Zero-latency로 처리함을 실증. 성공 기준(원한 필드 확보)·자동 재학습·저장방식 4종을 갖춘
**자기 개선형 크롤러**로 성숙. **(v6.0) 이미지 필드·첫 실행 자동 부트스트랩·폴더 재편·공유 레시피
레지스트리**를 더해 비전공자 배포까지 갖췄고, **(v6.1) 공유 레시피를 '검증된 안내서 + 가이드
임포트'로 재설계**했다. **(v6.2) 한국어 소스 + 영어 오버레이 i18n**을 도입해 기본 표시 언어를
영어로 전환(설정에서 ko↔en 토글) — 국제 배포·공유 레지스트리 유통을 겨냥한다. 자동 테스트 235개.

### 1.3 정의 및 약어

| 용어 | 정의 |
|------|------|
| by-example | 결과로 보이는 값을 주면 추출 규칙을 역으로 알아내는 방식 |
| 레코드(row) | 목록에서 반복되는 한 항목(공고/상품/글/항공권/트렌드 카드 한 개) |
| 앵커(anchor) | 사용자가 **가장 먼저 입력한 값**. 이 값이 있는 레코드를 기준으로 묶음을 잡음 |
| 구조 시그니처 | 클래스·텍스트를 버리고 태그 중첩 모양만 남긴 지문(`row_sig`) |
| 구조 경로(path) | 레코드 루트 기준 `(태그, 같은태그내 index)` 의 상대 경로 |
| 레코드 마커 | `data-testid`/`role` 등 컴포넌트 경계 속성. 클래스가 균일한 SPA에서 레코드 식별 기준 |
| 자가 치유(Self-Heal) | 셀렉터가 깨졌을 때 구조 경로·마커·LLM으로 재배치하고 갱신 |
| 로드방식(load_method) | `auto`(정적/SSR) / `render`(헤드리스 렌더) / `chrome`(내 크롬 Save As) |
| 레시피 | 사이트별 추출 규칙 + 로드방식·대기·페이지를 담은 CSV(사람이 열람 가능) |
| 감사 로그 | `_runs.csv` — 실행 한 건씩 성공/실패·번호·방식을 기록(재현·감사용) |
| replay | 과거 성공 크롤링을 번호로 골라 입력 없이 일괄 재현하는 도구 |
| Save As 수집 | 내 진짜 크롬으로 페이지를 열고 Ctrl+S로 로컬 저장 → 그 HTML을 파싱(로그인 세션 사용) |
| MDR | 반복 데이터 레코드 자동 탐지(Mining Data Records) 계열 기법 |
| LM Studio | 로컬 PC에서 LLM을 실행하는 OpenAI 호환 서버 |
| 체인 크롤링 | 부모 CSV의 링크 열 → 각 상세페이지를 다시 크롤(2단계). 상세=페이지당 레코드 1개 |
| 단일 레코드(single_record) | 상세페이지처럼 반복(≥2)이 없고 페이지당 레코드 1개인 모드. 반복 마커 미저장 |
| URL 정련 | 원본 링크 1개를 사용자가 고친 예시로 비교해 '지운 부분'(꼬리/감싼괄호/앞머리)을 열 전체에 적용 |
| iframe 인라인 | 상세 본문이 iframe 안에 있을 때(예: 채용정보 사이트 공고) 그 문서를 바깥 DOM에 끼워 넣어 잡음 |
| 본문 앵커(body:) | 본문에서 보이는 한 줄만 주면 그 줄이 든 컨테이너 전체를 필드로 잡는 지정 방식 |
| 미디어 폴백 | 본문이 텍스트가 아니라 이미지/링크뿐이면 그 URL을 대신 추출(media_urls) |
| 저장 방식(save_mode) | 결과를 쌓는 방식: append(중복제외 추가·기본)/history(전량 누적)/overwrite(교체)/upsert(키 갱신) |
| 회차(batch) | replay 세션마다 1 증가하는 정수(`_runs.csv` 최대+1). '이번에 추가된 것'을 시간 대신 회차로 식별 |
| 자동 재학습(AUTO_HEAL) | 값싼 치유가 다 실패 시 save_as HTML을 LLM이 통째 분석해 스키마 재생성(설정 ON일 때만) |
| 지식 저널(heal_knowledge) | AI로 뚫린 케이스의 [원인 + 개선 프롬프트 + HTML 샘플]을 남기는 개발자용 실패 진단 기록 |
| 성공 기준(커버리지) | '사용자가 원한 필드를 실제로 가져왔는가'. 준 예시 값의 절반도 못 잡으면 실패로 판정(LLM 불요) |
| 차단 마커 | 값이 인증/로봇확인 특유 텍스트(인증/verify/captcha/robot…)로 도배되면 실제 데이터 아님으로 판정 |
| 파일 잠금 대기(safe_io) | 엑셀 등이 CSV를 잠갔으면(PermissionError) 풀릴 때까지 기다렸다 저장(기록 유실 방지) |
| 다국어(i18n) | 한국어를 소스로 두고 언어별 오버레이를 얹는 방식. `t("한국어원문")` → 번역 있으면 번역, 없으면 한국어 원문(무중단 폴백) |
| 소스 언어 / 표시 기본값 | 소스=`ko`(키·폴백 기준, 불변) / 표시 기본값=`en`(`.env` LANG 미설정 시 사용자에게 보이는 기본 언어) |

## 2. 전체 설명 (Overall Description)

### 2.1 작동 워크플로우 (★ 핵심)

```
[사용자: 사이트 URL 입력]  (필요시 끝에 --chrome / --wait N / --pages N 등 플래그)
        ↓
[1] DOM 로드 (사이트 성격에 맞춰 자동/지정)
     · auto   : 정적 fetch 우선 (대부분, ~1초)
     · render : JS-SPA(동영상 플랫폼 등) — 헤드리스 브라우저 렌더링 (--wait 로 스트리밍 대기)
     · chrome : 안티봇/로그인(이커머스·소셜/트렌드 사이트·항공·여행 검색) — 내 진짜 크롬으로 Ctrl+S 저장 후 파싱
     · 차단(block_reason) 자동 감지 시 auto→chrome 로 자동 전환, 이후 1페이지만 수집
        ↓
[사용자: 보이는 값 한 줄 입력]   예) "제목@#12,000@#서울 강남구"
        ↓
[2] 파서 역설계 (단계 폴백, 사이트별 하드코딩 없음)
     ① 결정적 매칭 : 앵커(첫 값)의 '모든 등장 위치'에서 위로 올라가, 모든 값을 담고
                     '반복(≥2)'하는 가장 안쪽 조상을 레코드로. (정렬탭·스크린리더 등
                     비-레코드 위치는 자연히 배제) — 빠름/무료
     ②-a 마커 우선 : 클래스가 균일한 SPA면 data-testid/role 반복 경계를 레코드로 사용
     ②-b 렌더링   : 값이 '보이는 정적 HTML'에 하나라도 없으면 JS 렌더 후 재시도
     ③ LLM 분석   : 값이 쪼개져/표현이 달라 못 찾으면 레코드 HTML을 LLM에 주고
                     '역할(컬럼)' 기준 매핑 + 필드명 부여
        ↓
[3] 추출 + 페이지네이션 + 중복제거
     · 레코드 탐지 : 마커(data-testid) > 구조 반복(MDR) 순. 링크가 카드 전체를 감싼
                     '조상 <a>'면 각 행에서 위로 올라가 그 카드 링크를 잡음
     · 페이지     : 기본 1페이지(안전). --pages N 지정 시 순회. 차단 사이트는 강제 1p
     · dedup      : 링크 필드 우선(없으면 값 전체), 페이지/재실행 간 중복 제거
        ↓
[4] 출력 / 영속화 / 감사
     · CSV 누적    : 사이트별 파일에 신규만 append (+ crawled_at)
     · 레시피 저장 : 추출규칙 + load_method + wait + pages + row_testid 를 CSV로
     · 감사 로그   : _runs.csv 에 site_no·성공/실패·방식 기록
     · HTML 보관   : new(이번)/old(마지막 성공) 두 벌 → 잘 되던 사이트 실패 시 비교
     · 품질 가드   : 결과가 대부분 빈 값이면 저장 스킵(좋은 레시피 보호), 자동 재시도 안 함
        ↓
[재실행/스케줄]
     · replay.py  : _runs.csv 의 성공 건을 번호로 골라 입력 없이 일괄 재현
     · schtasks   : replay 를 3시간마다 → 정기 자동 수집(시계열 축적)
     · 사이트가 난독화/구조변경하면 자가 치유로 셀렉터 갱신 + 레시피 자동 업데이트
```

### 2.3 사용자 특성
- **1차 사용자**: 셀렉터/HTML을 모르는 일반 실무자(채용·가격·시세·항공권·트렌드 모니터링).
  "본 값을 적는다"는 행위만으로 크롤러를 만든다. **구별되는 값을 먼저** 주면 정확도가 오른다.
- **운영자**: 정기 수집을 스케줄링·관리하는 담당자. `_runs.csv`·레시피만으로 무엇을 언제
  어떻게 수집했는지 파악.

### 2.4 제약 조건
- **lxml** 필수.
- **Playwright**(선택): JS 렌더링/무한스크롤 대응. 미설치 시 정적 사이트만 동작(CLI 안내).
- **pywin32**(선택, Windows): chrome Save As 자동화(win32com/win32gui/win32clipboard).
  미설치 시 안티봇/로그인 사이트 수집 불가(정적·렌더는 정상).
- **LM Studio**(선택): LLM 분석/판단/명명. 미연결 시 결정적·구조 휴리스틱으로 폴백.
- **chrome Save As 방식은 데스크톱 세션 필요**: 실제 크롬 창을 띄우고 키를 보내므로,
  스케줄 시 '로그온 상태에서만 실행(/it)'으로 등록해야 하고 화면 잠금 시 실패(감사 로그에
  fail 기록, 자동 재시도 없음).
- 비전 불필요(텍스트 HTML 기반). 로컬 LLM(현재 Qwen3 계열)로 충분.

### 2.5 가정 및 의존성
- 목록 페이지에 같은 구조의 레코드가 2개 이상 반복된다고 가정.
- 사용자가 입력한 예시 값은 **한 레코드에서 본 것**이라고 가정. 서로 다른 항목의 값을
  섞으면 엔진이 감지해 "어떤 값이 어긋났는지" 안내하고 재입력을 유도.
- 대상 사이트의 robots.txt·이용약관·요청 한도 준수는 운영자 책임(본 엔진은 안정적 수집
  수단을 제공하되, 차단 사이트는 **1페이지·자동 재시도 없음**으로 신호를 최소화).
- Save As 방식은 **사용자 본인 계정의 로그인 세션**을 쓰므로, 사용자가 볼 수 있는 범위의
  데이터만 대상으로 한다(민감 페이지 URL 지정은 사용자 책임).

## 3. 기능 요구사항 (Functional Requirements)

### FR-01: DOM 로드 (3종 로드방식 + 차단 자동 전환)
- **입력**: URL 또는 로컬 HTML 경로, (선택) 로드방식/대기/페이지 지정
- **출력**: 파싱된 DOM
- **조건**:
  - `auto`: 정적 fetch 우선(~1초). 링크가 거의 없으면 SPA로 보고 렌더.
  - `render`: 학습 때 렌더가 필요했던 사이트(동영상 플랫폼 등)는 처음부터 헤드리스 렌더. `--wait`
    로 결과 스트리밍 대기(networkidle 후 추가 settle).
  - `chrome`: 내 진짜 크롬으로 URL을 열고 Ctrl+S(Save As) 자동 집행 → 로컬 HTML 파싱.
    디버그 포트 미사용(Chrome 136+ 기본 프로필 디버그 차단과 무관), 실제 로그인/쿠키 세션 사용.
  - 차단(`block_reason`) 감지 시 auto→chrome 자동 전환. 사용자 `--chrome`는 레시피
    load_method보다 우선.

### FR-02: 예시 기반 필드 역설계 (by-example, 앵커=첫 값)
- **입력**: 레코드 한 개의 값들 (`@#` 구분, 숫자 쉼표 보존)
- **출력**: 레코드 루트 + 필드별 (구조 경로/마커, 셀렉터, 예시 힌트)
- **조건**: **사용자가 가장 먼저 입력한 값**을 앵커로 삼는다(규칙이 단순·예측 가능 —
  구별되는 값을 먼저 주면 그 레코드로 잡힘). 앵커가 정렬탭·스크린리더 등 여러 곳에 나와도,
  **모든 등장 위치**에서 grounding을 시도해 '반복 레코드를 이루는' 위치만 채택.
  **링크(URL) 값은 href 속성으로 매칭**(자손이 없으면 카드를 감싼 조상 `<a>`도 탐색).

### FR-03: 레코드 마커 기반 탐지 (균일 클래스 SPA)
- **조건**: 클래스가 페이지 전반에 균일해 구조 시그니처가 과다매칭되면(소셜/트렌드 사이트 등),
  반복되는 `data-testid`(또는 `role=row/listitem/article`)를 레코드 경계로 최우선 사용.
  앵커의 '가장 안쪽 반복 마커 조상'을 레코드로. 항목마다 고유한 testid는 반복검증(≥2)에
  걸려 저장되지 않음(안전).

### FR-04: 비정상 감지 시 단계 에스컬레이션 + 사용자 안내
- **조건**: 반복 레코드를 못 찾으면 실패로 보고 렌더/LLM으로 넘긴다. 특히 **값들이 서로
  다른 항목에서 온 경우**(mixed-rows)는 LLM도 못 고치므로, 어떤 값이 어긋났는지 짚어
  **사용자가 스스로 한 항목에서 재입력**하도록 안내(잘못된 grounding로 전 행이 같은 값이
  되는 사고 방지).

### FR-05: LLM 역할 기반 매핑 (결정적 실패 시)
- **입력**: 주 반복 레코드 1개의 HTML + 사용자 값
- **출력**: 각 값 ↔ 레코드 내 실제 텍스트 매핑 + 역할 필드명
- **조건**: 텍스트 유사성이 아니라 '같은 역할(컬럼)' 기준. 값이 페이지에서 사라져도
  직무명을 회사명과 혼동하지 않는다.

### FR-06: 반복 레코드 자동 탐지 (구조)
- **출력**: 페이지 전역 반복 레코드 집합 + 시그니처
- **조건**: 형제 그룹을 느슨한 시그니처(depth=2)로 전역 합산. 텍스트 대부분이 '더 흔한
  하위 단위'로 채워지면 래퍼로 보고 제외. 점수 = 반복수 × 평균 텍스트량. 비콘텐츠 태그 제외.

### FR-07: 자가 치유 (Self-Heal)
- **조건(우선순위)**: ① 마커(data-testid/role) → ② 유일 클래스 가속 → ③ 구조 경로(class
  무관) → ④ LLM 재배치. 갱신된 스키마를 레시피에 자동 재저장.

### FR-08: 조상 링크 / 판매가 보강
- **조건**:
  - 카드 전체가 `<a href>`로 감싸진 경우(항공·여행 검색 등) 링크가 레코드 자손이 아니라
    조상 → 센티넬 경로로 저장, 추출 때 각 행에서 위로 올라가 그 카드 링크를 잡음.
  - 한 레코드에 가격류 숫자가 여럿(정상가/할인가/단가/적립/쿠폰)이면 문맥으로 걸러
    '실제 판매가'를 고름(쇼핑 카드).

### FR-09: 렌더 판정 (요소별, 보이는 텍스트 기준)
- **조건**: 결정적 매칭 실패 시, 각 텍스트 값이 **'보이는 정적 HTML'**(script/style/template
  제외)에 있는지 개별 확인. **하나라도 없으면** JS로 그려지는 것으로 보고 렌더 후 재시도.
  (text_content가 `<script>` 안 JSON까지 포함해 오판하던 문제 해결.) 학습 때 렌더가
  필요했으면 레시피 load_method=render로 기록.

### FR-10: 페이지네이션 + 중복 제거
- **조건**: 기본 **1페이지**(안전·봇신호 최소). `--pages N` 지정 또는 레시피값이 있으면 순회.
  차단 사이트는 값과 무관하게 1페이지. dedup은 변별력 있는 링크 우선, 가짜 링크는 키에서 제외.

### FR-11: 무한스크롤 대응
- **조건**: `--scroll` 시 헤드리스 브라우저로 바닥까지 반복 스크롤(높이 정체 시 종료).

### FR-12: CSV 누적 저장 + 품질 가드
- **출력**: `output/<사이트>_<해시>.csv`
- **조건**: 재실행 시 append, 기본 dedup(신규만), `crawled_at` 부가, utf-8-sig. 추출
  결과가 대부분 빈 값이면(자가치유가 엉뚱한 페이지에 끌려간 경우) **저장을 스킵**해 좋은
  레시피/CSV를 보호. **자동 재시도 없음**(차단 사이트에 신호를 더 보내지 않음).

### FR-13: 레시피 저장/재현 (CSV)
- **출력**: `recipes/<사이트>_<해시>.csv`
  (meta: url·row_tag·row_cls·row_css·row_signature·**row_testid·load_method·wait·pages** /
  field: 이름·tag·class·attr·경로·힌트)
- **조건**: 파일이 있으면 입력 없이 재현. **값이 아니라 '참조 대상'(컬럼 위치) + 사이트
  성격(로드방식·대기·페이지)** 을 기억 → 다음 실행이 같은 방식으로 재현. 자가 치유로 바뀐
  셀렉터 자동 갱신. (엑셀 레시피는 `--recipe`로 병행 지원.)

### FR-14: 감사 로그 (_runs.csv) + 재현(replay)
- **출력**: `output/_runs.csv` — `site_no, crawled_at, status, target, load_method,
  parse_method, input_example, fields, n_records, result_csv, recipe_csv, url_col`
  (**url_col**은 v4.0 체인용 — 부모 CSV의 어느 링크 열을 따라갔는지)
- **조건**:
  - 실행 한 건씩 성공/실패를 기록. **site_no**는 target 최초 등장 순의 고정 번호라, 파일만
    봐도 replay의 몇 번인지 안다(둘이 항상 일치). **체인 실행은 부모 밑 `P-k`**(→ FR-24).
  - `replay.py`가 성공 건을 사이트별 최신으로 모아 번호로 보여주고, 고른 번호를 cli에
    '주입'해 입력 없이 일괄 재현(외부 for문 구조 → 윈도우 스케줄러 연동 용이).

### FR-15: 프롬프트 플래그 파싱 / 재현 안전성
- **조건**: PowerShell에서 `&`가 많은 URL을 따옴표로 감싸기 번거로울 때, 대화형 프롬프트에
  `URL --chrome --wait 35 --pages 3`처럼 **끝에 아무 플래그나** 붙여도 떼어내 적용.
  단, `--rediscover`(일회성 레시피 삭제)는 **target에 절대 남기지 않음** → replay/스케줄러가
  매번 레시피를 지우는 '캐시 초기화 루프' 방지(strip이 저장/기록보다 먼저 실행).

### FR-16: HTML old/new 두 벌 보관
- **조건**: 수집한 HTML을 `output/saved/<사이트>.html`(new)에 저장하고, 추출이 정상이면
  `<사이트>.old.html`(마지막 성공본)로 승격. **잘 되던 사이트가 실패하면 old(성공)/new(실패)
  를 나란히 두고 원인 비교** 가능(가드가 두 경로를 안내).

## 3-b. 체인 크롤링 기능 요구사항 (v4.0 신규, 목록 CSV → 상세페이지)

> 1차 수집 결과(목록 CSV)의 링크 열을 따라 각 상세페이지를 **단일 레코드**로 2차 수집한다.
> `cli.run_chain_crawl()` 진입점. target 이 실제 `.csv` 파일이면(`_looks_like_csv`) 자동 진입,
> 일반 URL/HTML 단일 페이지 흐름과 완전히 분리된다.

### FR-17: 목록 CSV 입력 + URL 컬럼 지정
- **입력**: 목록 CSV 경로, (선택) `--url-col`
- **출력**: 순회할 상세 URL 목록(행 순서 유지)
- **조건**: `.csv` 감지 시 체인 모드 진입. 컬럼 목록을 보여주고 URL 열을 받되, 미지정 시
  `_guess_url_col`(우선순위 `*_url` > `url`/`링크` > `url` 포함)로 자동 추정해 Enter 기본값 제시.

### FR-18: URL 정련 by-example (열 전체 일괄)
- **입력**: 1행 원본 URL + 사용자가 정련한 예시 URL
- **출력**: 열 전체에 적용할 정련 함수(`_derive_url_cleaner`)
- **조건**: 정련한 값이 원본의 '연속 부분 문자열'이면 앞/뒤로 지운 리터럴을 전 행에서 제거
  (**꼬리 제거** `&src=..` / **감싸기 제거** `(...)` / **앞머리 제거** 모두 커버). 지운 꼬리가
  쿼리(`?..`/`&..`)면 **파라미터 이름 기준 제거**로 업그레이드(값이 행마다 달라도 처리).
  Enter 시 원본 유지. 부분문자열이 아니면 규칙 감지 실패로 경고 후 원본 유지.

### FR-19: 단일 레코드(상세) 스키마 생성 — 세 가지 지정 방식
- **입력**: 첫 상세페이지 DOM + 사용자 지정
- **출력**: `single_record=True` 스키마
- **조건**:
  - **(A) by-example**: `select_by_example(single=True)` → `locate_single_record`가 '모든 값을
    담는 가장 안쪽 컨테이너'를 반복검증 없이 레코드로 잡음.
  - **(B) 본문 앵커(`본문:`/`body:`)**: `_locate_body_container`가 그 한 줄을 담은 가장 작은
    요소를 찾아, iframe 인라인 경계(`data-inlined-iframe`)까지 올라가 본문 전체를(없으면
    '페이지 전체로 번지기 직전의 가장 큰 블록') 하나의 필드로.
  - **(C) CSS 셀렉터(`css:`)**: `_find_by_css`(`tag`/`.class`/`#id`/`tag.class`/`tag#id`, 뒤에
    `@attr`)로 노드를 찾아 필드화. 큰 컨테이너 지정 시 `link_split=False`로 자동 `_url` 미부착.
  - 세 방식 모두 문서 루트를 단일 레코드로 삼아 필드 위치를 **구조 경로**로 저장 → 같은
    템플릿의 다른 상세페이지에서 재현.

### FR-20: iframe 자동 인라인
- **조건**: 상세페이지 로드 후 `_inline_iframes`가 http(s) src iframe(광고/추적 호스트 제외,
  최대 N개)을 직접 fetch해 그 `<body>`를 iframe 자리에 `<div data-inlined-iframe>`로 끼운다.
  채용정보 사이트 공고처럼 본문이 iframe(예: `상세본문.asp`) 안에 있어 바깥 페이지가 '빈 껍데기'인
  경우에도 by-example/본문/CSS 지정이 본문을 실제로 찾게 한다. (사이트 하드코딩 없음.)

### FR-21: 상세 추출 — 고정 스키마 + 미디어 폴백 (★ 오염 방지)
- **조건**: `_extract_single`은 **스키마를 고정으로 다룬다(페이지별 자가치유로 변경·persist
  하지 않는다)**. 체인은 독립된 여러 페이지를 도는데, 어떤 페이지의 특이 케이스(예: 회사소개가
  텍스트가 아니라 이미지)로 필드 셀렉터를 재배치하면 그 변경이 뒤따르는 정상 페이지 전부를
  오염시키기 때문이다. 텍스트가 비면 그 페이지 **한정**으로 `media_urls`가 이미지 src(들),
  없으면 실제 링크 href(들)를 대신 뽑는다. `_find_single_root`는 컨테이너를 '텍스트 유무'가
  아니라 '노드를 찾았는가'로 평가(이미지 본문 페이지 대응).

### FR-22: 예의 있는 랜덤 대기 + 별도 상세 CSV
- **조건**:
  - 상세 요청 사이에 `random.uniform`으로 대기. `--delay`는 **숫자 하나**(평균 초, 기본 15=
    대형 사이트 기준)이고 `_delay_bounds`가 그 값 주위 **±50% 랜덤**으로 변환(규칙적 패턴 회피).
    학습에 쓴 첫 페이지(재사용)·스킵 항목엔 대기 없음. 실행 시 추천 표(`DELAY_HINT`)를 출력.
  - 결과는 `output/<csv이름>_detail.csv`로 **따로** 저장(원본 CSV 불변). `url_col` 값을 join
    키로 dedup, `crawled_at` 부가.

### FR-23: 체인 레시피 저장 / 입력 없는 재현
- **출력**: `recipes/<목록csv이름>__<url_col>.csv` (`chain_recipe_path_for`)
- **조건**: 학습이 정상이면 단일 레코드 스키마 + 재현 메타(**chain·url_col·clean_url**)를 레시피로
  저장(`save_csv_recipe(extra_meta=…)`). 다음 실행부터 같은 (CSV, url_col)이면 **입력 없이 재현**:
  스키마 로드 + `clean_url`로 URL 정련 규칙 재도출 → 필드 재지정/값 입력 불필요. `--url-col`
  미지정이고 이 목록의 체인 레시피가 하나뿐이면 자동 선택(`_find_chain_recipes`). `--rediscover`로 재학습.

### FR-24: 체인 실행 기록 + 부모-자식 번호 (P-k)
- **출력**: `_runs.csv`에 체인 실행 append(`url_col` 컬럼 추가), 상세 CSV/레시피 경로 기록
- **조건**: `assign_run_numbers`가 번호를 계층적으로 부여 — 일반 target은 정수(최초 등장 순),
  **체인(target 이 .csv)은 부모 밑 `P-k`**. 부모는 **result_csv 의 basename**으로 매칭(폴더 이동에도
  강함). `replay.py`가 `6-1` 형태로 목록에 표시하고, 고르면 `cli.py <목록csv> --url-col <col>`로
  비대화형 재현(레시피 자동 로드). `replay.py 1 3 6-1`처럼 부모·자식을 섞어 지정 가능.

## 3-c. 자동 재학습·성공 가드·저장 방식·설정 (v5.0 신규)

> 값싼 결정적 엔진이 대부분을 처리한다는 전제는 유지하되, **'원한 필드를 실제로 가져왔는가'를
> 성공의 기준으로 못박고**, 실패 시 **탐지를 피하며(save_as+오프라인 LLM) 스스로 배우는** 최후
> 사다리를 추가한다. 진입점은 기존 `cli` 흐름에 통합(별도 모드 아님).

### FR-25: 성공 기준 다층 가드 (★ '원한 필드를 실제로 가져왔는가')
- **입력**: 사용자 예시 값 개수(원한 필드 수) / 추출 결과 행·값 / 레시피 형태
- **출력**: 유효(저장) / 무효(스킵·재학습 후보)
- **조건**(값싼 것부터, 하나라도 걸리면 무효):
  - ① **필드 커버리지**(`_coverage_ok`, 결정적·LLM 불요): 학습 단계에서 준 예시 값의 **절반도
    못 잡으면**(matched vs selections) 실패로 보고 중단 + "N개 중 M개만 찾음 → save_as 로 다시"
    안내. auto 가 봇으로 보여 빈/인증 페이지를 받은 전형을 여기서 잡아 **레시피·출력 오염 0**.
  - ② **차단 마커**(`_looks_like_block`, 결정적): 추출 값의 절반 이상이 인증/로봇확인 텍스트
    (`인증`/verify/verification/captcha/robot/人机/滑块/보안문자…)면 무효.
  - ③ **의미 형태**(`_semantic_ok`): 예시 값의 형태(num/url/date)와 대조해 '채워졌지만 엉뚱한'
    치유를 거부(강한 형태·표본 ≥3·일치 <20%일 때만, 보수적).
  - ④ **극소수 LLM 백스톱**(`looks_like_real_records`): 행 **≤2**일 때만 LLM에 REAL/BLOCK 한 단어
    확인. **명시적 BLOCK일 때만** 무효(None·미연결·REAL=통과 → 못 믿을 때 정상 데이터를 막지 않음).

### FR-26: 자동 재학습 (save_as + 오프라인 LLM, 탐지 회피)
- **입력**: 무효 판정된 대상 + 기존 스키마(원한 필드명)
- **출력**: 재생성 스키마 + 재추출 결과 (채택 또는 롤백)
- **조건**:
  - **설정 `AUTO_HEAL` ON일 때만** 작동(기본 OFF = 기존 동작 불변). OFF면 "값싼 방법까지만
    분석했다"고 안내 후 중단.
  - 라이브 사이트엔 **save_as 1회만**(재fetch 금지) → 로컬 HTML 확보. 오프라인 LLM이 그 HTML을
    통째 분석해 **기존 필드명을 보존**하며 반복 레코드·필드 위치를 재발견(`recalibrate` →
    `discover_structure`). engine 은 **주입 훅**(`set_structure_discoverer`)만 호출(LLM-free 유지).
  - 재추출 후 **FR-25 가드로 재검증** → 통과 시 옛 레시피 백업 후 채택, 실패 시 **스키마 롤백**
    (`try_auto_heal`)으로 기존 레시피 보호. 성공 시 대화형으로 "이 사이트 로드=save_as/auto?" 확인.

### FR-27: 실패 진단 저널 (heal_knowledge — 개발자용, 엔진 자기개선)
- **출력**: `recipes/_heal_hints.json`(활성) / `_heal_resolved.json`(해결 로그)
- **조건**: AI로만 뚫린 케이스를 큐(field/site/example/source + 위치)로 기록하되, **왜 값싼
  휴리스틱이 놓쳤는지 + 어떻게 고칠지의 개발자용 개선 프롬프트**(`improvement_brief`)와 **문제
  HTML 샘플**을 함께 남긴다(개발자가 샘플로 프롬프트의 타당성을 검증 후 반영). (site,field,path)로
  dedup(최신 우선), site 우선 조회. **고치면 `resolve`가 사건을 해결 로그에 남기고 활성 저널에서
  삭제**(활성 저널엔 미해결만). 목적: 레시피는 사이트별 캐시, 저널은 **전 사이트 대상 엔진 개선의
  근거** → LLM 의존을 구조적으로 감소.

### FR-28: 저장 방식 4종 + 회차(batch)
- **입력**: 저장 방식(대화형/설정 기본/레시피값), 결과 행
- **출력**: `output/<사이트>.csv`(+ `회차`, `crawled_at`)
- **조건**:
  - 4종(`save_csv(mode=…)`): `append`(중복 제외 추가·기본) / `history`(전량 누적, 회차로 스냅샷) /
    `overwrite`(전량 교체) / `upsert`(키로 제자리 갱신). 저장 방식은 레시피 meta에 기록→replay 재현,
    `replay --list`에 `[덮어쓰기]` 등 한국어 라벨 표시.
  - **회차**: 상태 파일 없이 `runlog.next_batch()`=`_runs.csv` 최대+1. 한 replay 세션이 공유
    (체인 20초 대기로 시간이 흔들려도 회차는 안정). history=append+회차 → 별도 버전관리 불요.
  - **합집합 헤더**(기존 컬럼 보존 + 신규만 덧붙임): 레시피 필드명 ≠ 기존 output 컬럼명이어도
    옛 데이터 손실 0(이커머스/동영상 플랫폼 실복사본으로 무손실 검증).

### FR-29: 설정(.env) + 대화형 단일화
- **조건**:
  - `.env` 키: `AUTO_HEAL`(심층 재학습), `DEFAULT_SAVE_MODE`, `DEFAULT_LOAD_METHOD`(+ 기존 `LLM_*`).
    `services.llm_service.get_flag`로 읽고, `crawl_config`(leaf)가 기본 저장/로드 방식을 제공.
  - **start.py = 얇은 런처**(crawl은 `cli` 실행만). 실행 안내(로드/저장/주소/재학습)는 **cli
    한 곳**(`_interactive_setup`)에 통합 — 예전 start 자체 안내가 cli와 분기 갈려 'start 경로에선
    안 뜨는' 혼란을 제거. Enter(기본)도 **설정된 기본값을 일관되게 적용**(레시피는 설정 기준으로 작동).
  - 설정 메뉴: `LLM 공급자`(로컬 Qwen↔클라우드, `d`=심층 재학습 토글) / `레시피 기본 설정`
    (기본 저장·로드 방식) / `정합성 점검`. `.env` 쓰기는 **키 보존**(`_set_env`).

### FR-30: 파일 잠금 대기 (safe_io)
- **조건**: 엑셀 등이 CSV를 열어 잠갔으면(`PermissionError`) 풀릴 때까지 기다렸다 저장
  (`safe_io.open_when_writable`). runlog/output/레시피 쓰기 지점이 경유 → "엑셀 켜져 있어 기록
  못 함"으로 실행 결과가 유실되는 구멍 제거.

## 3-d. 이미지 필드 · 배포 · 공유 레시피 (v6.0 신규)

> 값싼 결정적 엔진(§3)은 그대로 두고, **결과물의 완성도(이미지)·배포성(비전공자 첫 실행)·유통(레시피 공유)**을 더한다.
> 브랜드명은 **Sovereign-Scraper**(데이터 주권 — 로컬 우선·자가 치유·내 레시피/데이터 소유·마스킹 공유).

### FR-31: 이미지 필드 — 구조 매칭 + 아카이브
- **문제**: 이미지는 URL로 매칭 불가 — save_as 스냅샷은 `_files/`로 로컬화(원격 URL 소실), 라이브 렌더는 CDN
  토큰이 매번 바뀜(뉴스포털 토큰형·이미지 플랫폼 사이즈변형). → **레코드마다 '구조'로 대표 이미지 선정**(`find_record_image`):
  1×1/`data:` 잡티 제외 → 예시 힌트 토큰(≥12자) 일치 → **최대 면적** → 첫 `<img>`. 시각적 피커(Shift+클릭)는
  화면에서 가장 큰 보이는 이미지를 잡는다. 타이핑된 이미지 URL(`.jpg` 등)은 링크가 아니라 이미지 필드로 인식.
- **아카이브**(`image_archive`, leaf): 원격 다운로드 / 로컬 복사, 확장자 추론(Content-Type → 매직바이트),
  URL 기준 dedup, 핫링크 보호 사이트(이미지 플랫폼 이미지 CDN)엔 Referer. 결과 CSV엔 **원격 URL과 오프라인 사본
  (`<필드>_file`) 둘 다** 남긴다(스냅샷 축적 비전). 이미지 경로는 루트 기준 상대 저장(폴더 이동 대응), 저장 방식
  (history=회차 폴더)은 CSV 규칙을 모방. (lazy-loading #10 이후 미로드는 알려진 한계 → 향후.)

### FR-32: 첫 실행 부트스트랩 (.venv 자동 · 비전공자용)
- **조건**: 배포 대상이 웹 요소를 눈으로 고르는 비전공자 → pip를 몰라도 되게. 첫 실행 시 **"가상환경(.venv)
  만들까요?(권장)"** 를 한 번 물어 선택을 기억(`.shc_bootstrap.json`), `.venv` 생성 + **의존성 자동설치**
  (`pip -r requirements.txt` + `playwright install chromium`) 후 **venv 파이썬으로 재실행**. 이후엔 자동으로 venv 사용.
- **배선**: 진입점 5개(start/cli/replay/capabilities/doctor) 최상단 `bootstrap.ensure_env()`(stdlib only, leaf).
  **안전장치**: 이미 어떤 venv 안이면 즉시 return(execv 무한루프 방지) · 비대화(비-tty)면 프롬프트 스킵 ·
  생성/설치/재실행 실패는 경고 후 현재 파이썬 폴백 · `SHC_NO_BOOTSTRAP=1` 우회 · 테스트 러너 자동 스킵 ·
  `if __name__=="__main__"` 가드로 `import cli` 등엔 미발동.

### FR-33: 폴더 구조 재편 + 경로 이식성
- **구조**: 루트(front)엔 사용자가 쓰는 것만 — `start.py`/`start.bat`·`cli.py`·`replay.py`·`output/`·`recipes/`·
  `requirements.txt`·`README`·`.env`. 나머지 전부 **`_internal/`**(engine·locators·crawlers·core·services·tests·docs 등).
  front 3개는 최상단에서 `sys.path` 에 `_internal` 추가(shim) → 기존 bare import 유지(코드 대량수정 없음).
- **앵커 보정**: 내부 모듈이 '자기 위치'가 아니라 '루트' 기준으로 `output/recipes/.env` 를 찾도록 깊이 보정
  (paths·bootstrap·crawl_config = 루트, services/llm_service = 3단계 위). `output.py`(모듈)가 `output/`(폴더)보다 우선.
- **경로 이식성**: 데이터 파일에 박히는 절대경로(레시피/`_runs.csv`/이미지)를 **루트 기준 상대**로 저장하고
  읽을 때 재-앵커(`paths.rel_to_root`/`abs_from_root`, 구버전 절대경로도 마커로 재해석) → **폴더를 옮기거나
  배포해도 그대로 재작동**. 테스트 실행 경로: `python _internal/tests/run_tests.py`.

### FR-34: 공유 레시피 레지스트리 (받기=HTTPS / 올리기=브라우저 PR)
- **원칙**: **받기는 쉽게(무인증·무 git), 올리기는 사람 검수(프라이버시)**. 레시피엔 크롤 URL(=검색어)이 박히므로
  자동 push 금지. git 라이브러리 내장은 보류(읽기엔 불필요, 쓰기 인증/프라이버시는 그대로).
- **레지스트리** = 공개 GitHub repo 하나: `index.json`(카탈로그) + `recipes/*.csv`(마스킹본).
- **받기**(`core/recipe_registry`, leaf·urllib): `fetch_index`(index.json) → `search`(로컬 키워드) → `download_recipe`
  → **`recipe_share.retarget_recipe`**(레시피 `meta.url` 을 '내 URL'로 재지정) → `recipes/` 슬롯에 설치 → 내 URL 크롤 시
  자동 로드. ★재지정 필수 이유: `paths._url_key` 가 **쿼리를 포함**해 슬롯을 매칭하므로, 마스킹 URL 그대로면 슬롯 불일치.
- **올리기**: 기존 마스킹 export(`replay --export-recipe`) 후 `share_page_url`(`<repo>/upload/main/recipes`)을
  `webbrowser.open` → 사람이 검수해 PR 제출. 설정 `.env` `RECIPE_REGISTRY_RAW`/`_WEB`(미설정=`OWNER` placeholder → 안내만).
- **메뉴**: `4. 레시피` → `1. 읽어들이기` / `2. 공유하기` / `3. 온라인에서 찾기`. (개발자 도구 doctor·역량매트릭스는 `5. 설정` 안으로 이동.)

### FR-35: 공유 레시피 재설계 — 검증된 필드맵 안내서 + 가이드 임포트 (v6.1)
- **문제(사용자 발견)**: 공유받은 레시피 파일을 `recipes/` 에 넣어도 개인 기록인 `_runs.csv` 에 실행 이력이 없어
  replay 목록에 안 뜨고 "무슨 URL·필드로 돌릴지 모르는" **고아 상태**. + export/다운로드가 `recipes/shared/` 한 곳에 혼재.
- **관점 전환**: 공유 = *파일 이식*이 아니라 **검증된 필드맵 안내서**. 받으면 → 내 URL 로 데려가 → **그대로 적용**
  (타인이 검증한 셀렉터 재사용)하거나 **처음부터 새로** 만든다. 실행이 `_runs` 기록+내 레시피를 남겨 고아를 없앤다.
- **폴더 분리**(`paths` 일원화): `recipes/shared/outbox`(내가 보낼 마스킹본)·`recipes/shared/inbox`(받은 것).
  `SHARED_DIR`/`OUTBOX_DIR`/`INBOX_DIR`, `inbox_dir()`/`outbox_dir()`. export→outbox, 레지스트리 다운로드→inbox.
- **자기설명 이름** `paths.share_label(target, fields)` = `사이트_필드1_필드2…`(예: `메일_제목_내용`).
  export 시 `공유 이름 [Enter=기본]:` — Enter=자동 생성 / 직접 입력 가능. 받은 사람이 **파일명만 봐도** 사이트·필드를 안다.
- **읽어들이기**(`start._import_recipe_file`): inbox 목록 → 선택 → **매니페스트(사이트·필드) 표시** → 내 URL 입력 →
  **Enter=검증본 그대로 `retarget`+설치+실행 / 아무거나 입력=처음부터 새로 학습**(필드 단위 편집은 단순화로 미채택).
  실측: 채용정보 사이트 공유본을 받아 → 읽어들이기 → 그대로 적용 → `_runs status=success` 새 기록 + replay 재현가능 확인.
- **매니페스트에 예시 주소 표시**: 읽어들이기 때 도메인만이 아니라 **마스킹된 전체 URL**(`…?searchword=EXAMPLE`)을
  보여줘 '어느 페이지·어떤 파라미터'인지 알린다(마스킹으로 검색어는 이미 지워졌으니 안전). 큰 사이트에서 "도메인만" 던지던 무책임 해소.
- **마스킹 강건화(v6.1, 공개 감사 중 발견)**: `mask_url` 이 쿼리값이 숫자면 길이 무관 보존해 **긴 숫자 ID**(체인
  `clean_url` 의 공고번호 등)가 샜다 → **5자리 미만 숫자만 보존, ≥5자리 숫자는 ID 로 보고 마스킹**(경로 세그먼트 규칙과 동일).
  `page=2`·`adultsv2=1` 등 구조값은 유지. 회귀 테스트 추가. ★공개 전 마스킹 감사(outbox 전수 스캔: 예시값·이메일·검색어·긴숫자ID)를 절차화.

## 3-e. 다국어(i18n) 지원 (v6.2 신규)

> 브랜드명 Sovereign-Scraper 부터 공유 레지스트리(§FR-34)까지 국제 배포·유통을 겨냥해온 흐름의
> 자연스러운 다음 단계. **기존 동작 0 변경**(§4.7의 기능 동결 원칙과 동일하게 적용) —
> 모든 사용자 문구를 그대로 두되 **표시 계층만** `t()`로 감싼다.

### FR-36: 한국어 소스 + 영어 오버레이 i18n (전 스크립트)
- **입력**: 한국어 원문 문자열(코드에 그대로 남음) + `.env`의 `LANG`
- **출력**: 현재 언어로 표시되는 문구 (번역 없으면 한국어 그대로)
- **조건**:
  - `_internal/i18n.py`(leaf, **표준 라이브러리만**, engine/cli/start 어디서도 최상위로 import 가능):
    `t(text, **kw)` — 현재 언어 번역표에서 `text`를 찾아 있으면 번역, **없으면 한국어 원문을 그대로
    반환**(폴백). 이 폴백 덕에 문자열을 하나씩 번역해도 앱이 깨지지 않는다("한국어로 완성 후
    언어를 확장"하는 점진적 접근, §슬라이스 1~8).
  - **언어 결정**: 소스 언어 `DEFAULT_LANG="ko"`(변경 없음, 번역표의 키 기준)와 **표시 기본값
    `DISPLAY_DEFAULT="en"`**을 분리한다. 루트 `.env`에 `LANG`이 있으면 그 값, 없으면 배포 기본인
    영어로 표시된다. `SUPPORTED=("ko","en")`.
  - **번역표**: `_internal/i18n/en.json` = `{"한국어원문": "영어번역", …}`, 현재 **428항목**. 언어
    추가 시 `SUPPORTED`에 코드 추가 + `<lang>.json` 하나만 있으면 된다(엔진 코드 변경 불필요).
  - **적용 범위**: 메뉴(start.py 전체 서브메뉴: 크롤/체인/replay/레시피/설정) · cli.py(대화형 프롬프트·
    정상경로 진행 메시지·실패/진단/가드 안내) · replay.py(목록·모드 라벨) · chain.py(체인 크롤 메시지) ·
    autoheal.py(치유 로그) · loader.py(SPA/차단 경고) · doctor.py(정합성 점검) · capabilities.py ·
    services/llm_service.py(연결 테스트) · crawlers/picker.py · build_registry_index.py · demo.py ·
    bootstrap.py(첫 실행 부트스트랩 — venv 존재 전에도 안전해야 하므로 i18n 자체가 stdlib-only leaf인
    것이 필수 전제) · safe_io.py(파일 잠금 대기 메시지). **검증 시점 기준 미번역 사용자 print/input 0줄.**
  - **런타임 전환**: `set_lang(lang)`으로 즉시 전환(캐시 갱신), `current_lang()`은 지연 초기화(최초 호출
    시 `.env` 읽음). 설정 메뉴(`5. 설정`)에서 ko↔en 토글 가능(쓰기는 `_set_env`가 다른 키 보존).
  - **품질 보증**: `test_i18n.py`(폴백/오버레이/포맷팅(`{kw}` 치환)/메뉴 커버리지/leaf 규율)로 회귀 방지.
    실스모크: 채용정보 사이트 실제 재현이 기본(영어) 언어로 완주(`status=success`) 확인.

## 4. 비기능 요구사항 (Non-Functional Requirements)

### 4.1 성능
- 정적 사이트 1페이지: ~1–2초(LLM 불필요 시). 렌더/Save As는 사이트 로딩·대기에 비례.
- LLM은 결정적 매칭 실패 시에만. 페이지네이션은 패턴 학습 후 무호출.

### 4.2 견고성 / 자가 치유
- 클래스 난독화 무관(구조 경로/마커 anchor). 구조 변경 시 LLM이 변경분만 패치.
- 앵커가 여러 곳에 나와도 '반복 레코드를 이루는 위치'만 채택(정렬탭·스크린리더 노이즈 방어).
- lxml 프록시 id() 재사용 함정 회피.

### 4.3 폴백 안정성 (Graceful Degradation)
- LLM 미연결 → 휴리스틱/구조로 폴백. Playwright 미설치 → 정적만 + 안내. pywin32 미설치 →
  안티봇/로그인 제외 정상. 클립보드 실패 시 → 타이핑 폴백.
- 로드 실패/환각 URL/무한루프/차단 페이지 페이지네이션 가드 내장.

### 4.4 운영성 / 재현성
- 사이트별 규칙·성격을 코드가 아니라 **레시피(CSV)**로 외부화(비개발자 열람).
- **감사 로그 + replay + 스케줄러 = 입력 없는 정기 자동 수집**. site_no로 파일·재현 번호 일치.
- **캐시 후 분석 원칙**: 받아둔 HTML로 오프라인 반복 분석 → 타겟에 추가 신호 0으로 디버깅·개선.

### 4.5 정직성/투명성
- 어떤 경로(결정적/마커/렌더/LLM)로 풀었는지, 로드방식·대기·페이지, 페이지별 신규 건수,
  셀렉터 갱신 내역, 성공/실패를 모두 로그·`_runs.csv`로 표시.

### 4.6 안전/예의 (Anti-Detection Etiquette)
- 차단 사이트는 **1페이지만**, **자동 재시도 없음**. Save As는 마우스 이동 0·키 입력 최소로
  봇 탐지 회피. 실패는 로그로 남겨 사람이 판단.
- **(v4.0) 체인 크롤링은 상세 요청 사이에 랜덤 대기**(기본 평균 15초, ±50%). 부모 CSV의
  링크를 연속으로 두드리면 특정 사이트에 부담/차단 신호가 되므로, 대형 사이트 기준을 기본값으로
  두고(빠르게 하려면 사용자가 `--delay`를 낮춤) 매 요청 간격을 무작위화해 규칙적 패턴을 피한다.

### 4.7 (v4.1) 유지보수성 / 모듈성 (Maintainability)
- **관심사 분리(SoC)**: 거대 단일 파일에서 **LLM·수집·데이터 계약**을 계층으로 분리(§5-b).
  `engine.py`는 이제 LLM·fetch·스키마 정의를 **직접 갖지 않고** 하위 계층에 위임한다.
- **의존성 단방향(아래로만)**: `cli/replay → engine → (llm_locators) → services·crawlers·core`.
  leaf 계층(services/crawlers/core)은 상위 모듈을 import 하지 않는다.
- **경계는 문서가 아니라 테스트로 강제**: engine LLM-free / crawlers·core leaf 규율을 **소스
  스캔 테스트**가 검사 → 누가 어기는 import를 넣으면 즉시 실패(주석의 노후화(stale) 문제 방지).
- **안전한 점진 리팩터**: 매 단계 '**테스트 그린 + 실사이트 1건 스모크**'를 통과할 때만 이동
  (기능 동결·동작 0 변경). 회귀 안전망으로 자동 테스트 유지(v4.1 37개 → v5.0 140개).

### 4.8 (v5.0) 정확성 계약 / 자기 개선 (Correctness & Self-Improvement)
- **성공의 정의를 명시화**: '값이 채워졌다'가 아니라 **'사용자가 원한 필드를 실제로 가져왔다'**.
  준 예시 값의 절반도 못 잡거나 값이 차단/인증 텍스트로 도배되면 성공이 아니며, 잘못된 성공으로
  좋은 레시피/출력을 덮지 않는다(오탐 저장 방지가 데이터 신뢰성의 핵심).
- **탐지 회피 우선의 재학습**: 라이브 사이트엔 save_as 1회만(재fetch 없음), 무거운 전체-HTML LLM
  분석은 **로컬 파일에만** 수행 → 사이트에서 스크립트 접근·반복 요청을 보지 못한다.
- **저비용 우선순위**: 추가 네트워크 요청 0인 값싼 방법(정적/구조/휴리스틱/커버리지·차단 가드)을
  **모두** 소진한 뒤에만 비싼 LLM 재학습을 쓴다(설정 ON일 때). 그래야 그 비싼 지식이 값지다.
- **자기 개선 루프**: 정답을 찾으면 레시피로 가벼워지되(사이트별), 뚫린 경위는 저널로 남겨
  **값싼 휴리스틱 자체(정적/동적/구조)를 개선**해 다음엔 LLM 없이 통과하게 만든다(개발자가 고치면
  resolve로 정리). 궁극 목표는 **LLM 의존을 시간이 지날수록 줄이는 것**.

### 4.9 (v6.0) AI-운용 가능성 / 결정적 진단 폐루프 (AI-Operability & Deterministic Diagnosis Loop)
> §4.7의 테스트 강제 경계와 §4.4의 '캐시 후 분석' 원칙이 결합해 **드러난(emergent) 속성**이다. 원래
> 이 안전망은 *사람이 cli를 안전하게 분할*하려고 설계했는데, 같은 성질이 **AI(또는 사람)가 결함을
> 오프라인·결정적으로 재현·수정·검증**하기에도 이상적임이 실전에서 확인됐다.

- **재현 기판(substrate)**: `output/saved/<site>.html`(+ `_files/`)로 **실패 순간의 DOM이 냉동 저장**되고,
  leaf 함수(주입 가능)·소스스캔 경계 테스트·높은 테스트:코드 비율(현재 8.9K:7.2K)이 갖춰져, 라이브
  사이트 없이 **내부 로직을 직접 호출해 결정적으로 재현**할 수 있다.
- **진단 폐루프(사이트 재접속 0)**: ① 이상한 output ↔ 저장 HTML 대조로 결함 국소화 → ② 원인 추론 →
  ③ 최소 수정(leaf 우선) → ④ **재현된 실패가 사라지고 + 전체 테스트가 그린**임을 기계적으로 확인 →
  ⑤ 행동을 골든 테스트로 못박음. 신뢰의 근거는 "동작한다"가 아니라 **재현 실패의 소멸 + 무회귀**다.
- **실증(v6.0)**: 웹메일 대화 메일의 '제목 밀림' 결함(개수 뱃지 span 삽입으로 index 시프트, 100행 중 13행)을
  `output_webmail_1.html` 하나로 재현→로케이터 개선(구조경로에 형제-유일 class 앵커 + 팬텀 형제 제거)→
  13→**0행** 검증까지, **웹메일에 추가 요청 0**·자동 테스트 221개 그린으로 완료(§FR-33 구조 이식성과 시너지).
- **정직한 경계(과신 금지)**: 스냅샷은 노후화하고(난독 class·CDN 토큰 회전), AI는 오진할 수 있으며,
  '무엇이 틀렸는지'는 여전히 사람이 가리켜야 한다. 따라서 이는 '자율 운용'이 아니라 **결정적 검증
  기판 위의 AI-보조 유지보수**다 — 안전성은 자동화 수준이 아니라 *검증의 결정성*에서 온다.

### 4.10 (v6.2) 국제화 / 무중단 번역 확장 (Internationalization & Non-Breaking Localization)
- **소스-폴백 분리**: 번역표에 없는 문자열은 예외 없이 한국어 원문으로 표시된다(§FR-36) — 번역이
  부분적이어도 앱이 깨지거나 빈 문구가 나오지 않는다. 번역은 점진적으로, 언제든 중단·재개 가능.
- **표시 계층에만 개입**: `t()`는 로그·CSV·레시피·`_runs.csv` 등 **데이터 계약에는 개입하지 않는다**
  (필드명·저장값은 여전히 한국어/원문 그대로) — 기존 레시피·자동화 스크립트·RTM(§8)이 영향받지 않는다.
- **leaf 규율 확장**: `i18n.py`는 표준 라이브러리만 쓰는 leaf라 `bootstrap.py`(venv 생성 이전)에서도
  안전하게 호출된다 — §4.7의 "경계는 테스트로 강제" 원칙이 다국어 계층에도 동일하게 적용된 사례.

## 5-b. 코드 아키텍처 & 계층 경계 (v4.1 모듈화)

> **원칙**: 기능은 그대로(동결), 구조만 안전하게 분리. 각 단계는 *테스트 그린 + 실사이트 스모크*를
> 통과할 때만 이동. 경계는 **주석이 아니라 테스트로 강제**한다(자세한 규율은 `CONTRIBUTING.md`).

### 계층 & 의존성 방향 (한 방향, 아래로만)

```
cli.py / replay.py            컨트롤러(오케스트레이션/엔트리포인트)
      │
      ▼
engine.py                     DOM 추출·자가치유 (LLM·fetch·스키마 정의를 직접 갖지 않음)
      │  └─ 의미 기반 재배치는 '주입된 훅'(set_relocator)만 호출 → LLM-FREE
      ▼
llm_locators.py               DOM+LLM 오케스트레이션(engine·service를 잇는 접착층)
      ▼
services/llm_service.py   ·   crawlers/{base,static,dynamic,chrome}   ·   core/schema.py   (전부 leaf)
      ▼
requests · lxml · playwright · pywin32 · openpyxl   (외부 라이브러리)
```

### 모듈 책임

| 계층/파일 | 책임 |
|-----------|------|
| `cli.py` / `replay.py` | 대화형/비대화형 흐름, 체인 크롤링, 감사로그·번호매기기, 재현 |
| `engine.py` | 구조 시그니처·반복탐지·by-example·자가치유·추출 (**LLM/수집/스키마 정의 없음**) |
| `llm_locators.py` | LLM 의미 기반 매핑/재배치/명명/다음페이지(엔진 헬퍼 + LLM 전송을 조합) |
| `services/llm_service.py` | LLM 전송(LM Studio, OpenAI 호환). 평범한 타입만 주고받는 leaf |
| `crawlers/base.py` | 공용 UA/헤더 + 안티봇 차단 감지(`block_reason`) |
| `crawlers/static.py` | 정적/SSR fetch(requests>urllib) |
| `crawlers/dynamic.py` | Playwright 렌더/무한스크롤/느린 SPA |
| `crawlers/chrome.py` | '진짜 크롬' 수집(CDP attach / Save As 자동, pywin32) |
| `core/schema.py` | 데이터 계약(`Schema`/`FieldSchema`) + 레시피 직렬화(순수 데이터) |
| `heal_knowledge.py` | (v5.0) 실패 진단 저널 leaf(record/hints_for/all_hints/resolve). stdlib+paths+safe_io만 |
| `crawl_config.py` | (v5.0) `.env`의 기본 저장/로드 방식 제공 leaf(stdlib os만) |
| `safe_io.py` | (v5.0) 파일 잠금 대기 저장 leaf(엑셀 잠금 시 재시도). 쓰기 지점이 공용 경유 |
| `runlog.py` | 감사 로그·회차(`next_batch`)·저장방식 라벨(`MODE_LABELS`) leaf |

### 강제되는 불변식(테스트가 지킴)

- **engine LLM-free**: `engine.py`가 `llm*`을 import하면 `test_engine_llm_free`가 실패.
  (v5.0) 자동 재학습의 구조 파악도 **주입 훅**(`set_structure_discoverer`/`_discover_structure`)으로만
  호출 — engine 은 여전히 LLM을 직접 갖지 않는다(같은 테스트가 신규 훅도 확인).
- **crawlers/core leaf**: 이 계층이 상위(engine/cli/…)를 import하면 해당 계층 테스트가 실패.
  (v5.0) `heal_knowledge`/`crawl_config`도 leaf 규율(상위 미import) 테스트로 강제.
- **주입 훅 배선**: cli가 `set_relocator`·`set_structure_discoverer`로 재배치자/구조탐색자를 주입했는지
  테스트가 확인(미주입 시 engine은 구조/휴리스틱만으로 정상 동작 = graceful).

## 6. 기존 웹 크롤링 방법과의 차이점 (★ 제안 핵심)

### 6.1 한눈 비교

| 항목 | 전통적 크롤러<br>(BeautifulSoup/Scrapy + CSS·XPath) | 상용 'AI 크롤러'<br>(클라우드 LLM 추출) | **본 엔진** |
|---|---|---|---|
| 규칙 작성 | 개발자가 사이트별 셀렉터 수작업 | 자연어 프롬프트 | **본 값 한 줄(by-example), 첫 값이 기준** |
| 클래스 난독화 | **파손**(재작성) | 매번 LLM 재해석 | **구조 경로/마커로 무관, 자동 유지** |
| 구조 변경 | **파손** | LLM 재해석(비용↑) | **LLM이 깨진 필드만 패치 후 갱신** |
| 클래스 균일 SPA(소셜/트렌드 사이트 등) | 셀렉터 특정 어려움 | 가능하나 비쌈 | **data-testid/role 마커로 정확 식별** |
| JS 렌더링(SPA) | 별도 렌더 파이프라인 수작업 | 서비스 의존 | **요소별 판정 후 자동 렌더, 레시피에 기록** |
| **안티봇(Akamai/DataDome)** | 대개 **차단**(우회 코드 필요) | 프록시/캡차 비용 | **내 크롬 세션 Save As로 우회, 1p·무재시도** |
| **로그인 필요 페이지** | 세션 쿠키 수작업 주입 | 자격증명 위탁(위험) | **내 브라우저 로그인 세션 그대로 재사용** |
| LLM 비용/지연 | 없음 | **매 페이지 호출** | **필요할 때만**, 페이지네이션 무호출 수렴 |
| 데이터 위치 | 사람이 지정 | LLM이 매번 | **코드가 도출(경로/마커), LLM은 의미만** |
| 비개발자 사용 | 어려움 | 가능하나 불안정/종속 | **가능 + 결과·규칙을 사람이 검증(CSV 레시피)** |
| 데이터 주권 | 로컬 | **외부 서버 전송** | **전부 로컬**(로컬 LLM, 내 브라우저) |
| 재현/운영 | 코드 버전관리 | 서비스 종속 | **레시피 + 감사로그 + replay + 스케줄러** |
| 사이트별 '성격' | 코드에 산재 | 없음 | **로드방식·대기·페이지를 레시피에 학습·재현** |
| **성공 판정** | 예외 안 나면 성공 | 뭔가 채우면 성공 | **'원한 필드를 실제로 가져왔는가'(커버리지·차단·의미·백스톱)** |
| **실패 시 학습** | 사람이 수정 | 매번 LLM 재추론 | **탐지 회피 재학습(save_as+오프라인 LLM) + 저널로 엔진 개선** |
| **저장 전략** | 수작업 | 서비스 종속 | **추가/중복제외/덮어쓰기/키갱신 4종 + 회차(합집합 헤더 무손실)** |

### 6.2 차별점 상세

**(1) 셀렉터가 아니라 '구조적 불변성/컴포넌트 경계'에 anchor.**
전통 크롤러는 `div.post-title` 클래스에 의존해 난독화/리뉴얼에 깨진다. 본 엔진은 클래스를
'있을 때만 쓰는 가속기'로 강등하고, 진짜 기준은 **태그 중첩 구조·상대 경로**이며, 클래스가
아예 균일한 SPA(소셜/트렌드 사이트)는 **data-testid/role 컴포넌트 경계**를 쓴다. 매 빌드 랜덤화돼도 유지.

**(2) LLM은 '의미'만, 위치 도출은 코드가.**
상용 AI 크롤러는 매 페이지를 LLM에 넣어 비싸고 느리고 불안정하다. 본 엔진은 결정적 매칭으로
대부분 처리하고, **LLM은 역할 매핑·다음 페이지 판단** 같은 '판단'에만 쓴 뒤 그로부터
셀렉터/URL 패턴을 학습해 이후엔 LLM 없이 진행한다.

**(3) programming-by-example — 비개발자가, 예측 가능하게 만든다.**
HTML 지식 없이 "본 값"만 주면 파서가 만들어진다. **첫 값이 기준**이라는 단순 규칙이라
"구별되는 값을 먼저 준다"는 직관으로 정확도를 통제할 수 있고, 결과 규칙은 **CSV 레시피**로
열어볼 수 있어 블랙박스가 아니다.

**(4) 안티봇·로그인 사이트를 '내 브라우저 세션'으로 정면 돌파.**
전통/상용 도구가 프록시·캡차·자격증명 위탁으로 씨름하는 영역을, 본 엔진은 **사용자가 이미
로그인한 진짜 크롬으로 페이지를 열어 Ctrl+S로 저장** 하는 방식으로 우회한다. 재인증도, 봇
탐지 위험(마우스 이동 0)도, 외부로의 자격증명 유출도 없다. Akamai(이커머스)·DataDome
(항공·여행 검색)·로그인 세션(소셜/트렌드 사이트)에서 실제 수집을 확인했다.

**(5) 사이트마다 다른 '수집 성격'을 학습·재현.**
같은 크롤러라도 사이트마다 최적 로드방식(정적/렌더/크롬)·로딩 대기·페이지 수가 다르다. 본
엔진은 이를 **레시피에 학습**해, 재현 시 target만 주면 그 사이트에 맞는 방식으로 알아서
로드한다(이커머스=chrome·1p, 동영상 플랫폼=render, 항공·여행 검색=chrome·wait 35).

**(6) 자가 치유 + 감사 + 재현 = 운영 가능한 크롤러.**
사이트가 바뀌면 전통 크롤러는 사람이 고치고, AI 크롤러는 매번 다시 추론한다. 본 엔진은
구조/마커로 버티다 바뀐 부분만 LLM이 한 번 고치고 **레시피를 자동 갱신**하며, 모든 실행을
`_runs.csv`에 기록하고 **replay + 스케줄러**로 입력 없이 정기 수집한다. 실패는 로그로
남기고 **자동 재시도하지 않아** 차단 사이트에 신호를 더 보내지 않는다.

**(7) 수집과 분석의 분리 — 받아둔 HTML로 오프라인 개선.**
받아온 HTML을 저장(old/new)해 두고 파싱·디버깅·규칙개선을 로컬 파일로 반복한다. 타겟에
재접속 0으로 원인을 분석·검증하므로, 안티봇 사이트를 디버깅하다 차단당하는 일이 없다.

**(8) (v5.0) 성공을 '원한 것을 가져왔는가'로 정의 — 오탐을 저장하지 않는다.**
상용 도구는 무언가만 채우면 '성공'으로 반환해, 인증창·빈 페이지를 데이터로 넘기기 쉽다. 본
엔진은 **사용자가 지정한 필드의 절반도 못 잡으면 실패**로 보고(결정적, LLM 불요), 차단 마커·
형태 불일치까지 다층으로 걸러 **잘못된 성공으로 좋은 레시피를 덮지 않는다.** 데이터 신뢰성은
'무엇을 뽑았나'만큼 '무엇을 저장하지 않았나'에서 온다.

**(9) (v5.0) 탐지를 피하며 스스로 배우고, 그 지식으로 엔진을 개선한다.**
값싼 방법이 다 실패하면 **save_as로 로컬에 받은 HTML만 오프라인 LLM에 먹여** 원한 필드를
재발견한다(라이브 추가접속 0). 게다가 뚫린 경위를 **개발자용 진단 저널**(개선 프롬프트 + HTML
샘플)로 남겨, 값싼 휴리스틱 자체를 개선해 **LLM 의존을 시간이 지날수록 줄인다.** '매번 LLM에
기대는' 상용 AI 크롤러와 정반대 방향이다.

### D. 알려진 한계 / 향후 과제
```
- 제목+메타가 형제로 쪼개진 테이블의 record-grouping 미구현
- 무한스크롤 사이트의 '가짜 다음 페이지' 오탐 여지(기본 pages=1로 회피)
- chrome Save As 방식은 데스크톱 로그온 세션 필요(무인 서버 부적합) — auto/render는 무인 가능
- 레시피 필드명이 LLM 명명이라 다듬음이 필요할 수 있음(사람이 CSV에서 수정 가능)
- robots.txt/요청 레이트 계층은 운영자 책임(엔진은 1p·무재시도로 신호 최소화)
- [속성 기반 추출 미구현] 복잡한 인증 웹앱(웹메일 등)은 부분 추출에 그침: 논리적 값
  (실제 발신자 이메일)이 텍스트가 아니라 속성(email/name/title)에 있고, 알림·스레드 행은
  표시 텍스트가 잘리거나('이름 on 소셜앱'→'이름 on.') 참여자가 합쳐진다. '보이는 텍스트=데이터'
  가 성립하는 사이트(검색/쇼핑/게시판)엔 완벽하나, 웹앱은 속성 기반 추출이 향후 과제.
  · (v5.0) 다만 **save_as + overwrite로 웹메일 받은편지함 99건을 Zero-latency 재현** 확인
    (로그인 세션 HTML을 결정적 매칭). 위 '속성 기반' 한계는 논리값(발신자 이메일 원문)에 한함.

- [v5.0 첫 학습 자동 재학습 미연결] AUTO_HEAL은 '기존 레시피 재추출 실패'에만 작동. 처음
  배우다 커버리지 미달로 막히면 커버리지 가드가 잡아 save_as 재시도를 '안내'만 한다(자동
  save_as 재시도는 향후 과제). 극소수 LLM 백스톱은 행 ≤2에서만 호출(비용 억제).
- [cli 비대화형 분리 여지] 성공 가드·자동 재학습이 cli에 누적 — `guards.py`/`autoheal.py`로
  추출해 cli를 오케스트레이터로 얇게 하는 리팩터가 다음 후보(§7 교훈 g의 연장).

### D-1. 활용 비전 / 프라이버시 (개인 데이터 주권)
```
잠재력: 내 로그인 세션을 재사용하므로, '내가 접근 가능한 모든 곳'의 데이터를 로컬 CSV로
        구조화할 수 있다 — 은행 거래내역→가계부, 메일→요약, 주문/구독 정리 등, 흩어진
        내 데이터를 사일로에서 내 로컬 통제로 되찾는 '개인 데이터 주권' 도구가 될 수 있다.
전제/가드: 반드시 '내 데이터'만. chrome Save As 는 내 세션이 있어야만 되므로 타인이
        재현 불가. 다만 강력한 만큼 아래 프라이버시 계층이 필수(현재 일부 수동, --private 예정):
        · saved/<site>.html 는 평문 → 민감 사이트는 파싱 후 삭제
        · recipes/·_runs.csv 는 git 추적 → '예시값'에 실제 내용이 들어가지 않게 마스킹
          (input_example·필드 example 힌트를 '[비공개]'로) — 공개 저장소 유출 방지
        · 민감 사이트는 스케줄 자동수집 대상에서 제외 권장
검증 사례: 웹메일 받은편지함 role=row 마커로 구조화(발신자/제목/시간). (v5.0) **save_as +
        overwrite로 99건 Zero-latency 재현**('내 편지는 누적 불필요' → 덮어쓰기). 커뮤니티 게시판
        50건 save_as 돌파. 단, 위 '속성 기반 추출' 한계로 일부 발신자 값이 표시 텍스트대로
        부분적. (분석 후 데이터 삭제 — saved/<site>.html 평문 주의)
```

```
